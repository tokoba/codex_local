# TUI ストリームチャンク処理

このドキュメントでは TUI におけるストリームのチャンク処理がどのように動作し、なぜこのように
実装されているかを説明します。

## 問題

ストリーミング出力は、1 ティックあたり 1 行のアニメーションが表示できる速度よりも速く到着する
ことがあります。到着速度がスパイクしてもコミット速度が一定のままだと、キューされた行が増加し、
画面に表示される出力が受信済みの出力に対して遅延することになります。

## 設計目標

- 通常の負荷では既存のベースライン動作を維持すること。
- バックログが溜まったときに表示の遅延を減らすこと。
- 出力の順序を安定して保つこと。
- 見た目がぎこちなくなる、突然の 1 フレーム分の一括フラッシュを避けること。
- ポリシーはトランスポート非依存とし、キューの状態のみに基づくように保つこと。

## 非目標

- このポリシーはアニメーションティックのスケジューリングは行わない。
- このポリシーは上流のソースの識別情報には依存しない。
- このポリシーはキューされた出力の並べ替えを行わない。

## ロジックの所在

- `codex-rs/tui/src/streaming/chunking.rs`
  - アダプティブなポリシー、モード遷移、およびドレインプランの選択。
- `codex-rs/tui/src/streaming/commit_tick.rs`
  - 各 commit ティックに対するオーケストレーション: スナップショット、決定、ドレイン、トレース。
- `codex-rs/tui/src/streaming/controller.rs`
  - commit ティックのオーケストレーションで使用されるキュー／ドレインのプリミティブ。
- `codex-rs/tui/src/chatwidget.rs`
  - commit ティックのオーケストレーションを呼び出し、UI のライフサイクルイベントを処理する
    統合ポイント。

## 実行時のフロー

各 commit ティックごとに次を行います:

1. アクティブな controller 全体にわたるキュースナップショットを構築する。
   - `queued_lines`: キューに溜まっている行数の合計。
   - `oldest_age`: 各 controller における最も古いキュー行の経過時間の最大値。
2. アダプティブポリシーに決定を問い合わせる。
   - 出力: 現在のモードとドレインプラン。
3. 各 controller にドレインプランを適用する。
4. 呼び出し元が挿入できるように、ドレインされた `HistoryCell` を出力する。
5. オブザーバビリティのためにトレースログを出力する。

`CatchUpOnly` のスコープ内では、ポリシーの状態は進行し続けるが、モードが現在
`CatchUp` でない限りドレインはスキップされる。

## モードと遷移

使用されるモードは 2 種類です:

- `Smooth`
  - ベースラインの動作: ベースラインの commit ティックごとに 1 行をドレインする。
  - ベースラインのティック間隔は現在
    `tui/src/app.rs:COMMIT_ANIMATION_TICK` から取得される (~8.3ms、約 120fps)。
- `CatchUp`
  - `Batch(queued_lines)` により、ティックごとに現在のバックログキューをドレインする。

モードの突入と離脱にはヒステリシスを用いる:

- キューの深さまたはキューの経過時間がエントリしきい値を超えたときに `CatchUp` に入る。
- 離脱には、ホールドウィンドウ (`EXIT_HOLD`) の間、深さと経過時間の両方がエグジットしきい値を
  下回っていることが必要。

これにより、負荷がしきい値付近を行き来する場合のモードの振動を防ぐ。

## 現在の実験的なチューニング値

以下は `streaming/chunking.rs` における現在の値と、`tui/src/app.rs` における
ベースラインの commit ティックです。これらは実験的な値であり、より多くのトレースデータが
集まるに従って変更される可能性があります。

- ベースラインの commit ティック: `~8.3ms` (`app.rs` の `COMMIT_ANIMATION_TICK`)
- Catch-up への突入:
  - `queued_lines >= 8` OR `oldest_age >= 120ms`
- Catch-up からの離脱の条件:
  - `queued_lines <= 2` AND `oldest_age <= 40ms`
- 離脱ホールド (`CatchUp -> Smooth`): `250ms`
- Catch-up 離脱後の再突入ホールド: `250ms`
- 深刻なバックログのしきい値:
  - `queued_lines >= 64` OR `oldest_age >= 300ms`

## ドレインプランニング

`Smooth` のとき、プランは常に `Single` となる。

`CatchUp` のとき、プランは `Batch(queued_lines)` であり、現在キューされているバックログを
ドレインして即時収束させる。

## この設計にした理由

この設計により、通常のアニメーションのセマンティクスを維持しつつ、バックログ時の挙動を
アダプティブにしている:

- 通常の負荷では、動作は従来どおりで安定している。
- 負荷が高いときには、順序を損なうことなくキューの経過時間を素早く短縮する。
- ヒステリシスにより、モードが急激に切り替わり続けることを防ぐ。

## 不変条件

- キューの順序は保持される。
- キューが空になると、ポリシーは `Smooth` にリセットされる。
- `CatchUp` は、低負荷の状態が一定期間続いた後にのみ抜ける。
- `CatchUp` 中は catch-up のドレインが即時に行われる。

## 可観測性

トレースイベントは commit ティックのオーケストレーションから出力される:

- `stream chunking commit tick`
  - `mode`, `queued_lines`, `oldest_queued_age_ms`, `drain_plan`,
    `has_controller`, `all_idle`
- `stream chunking mode transition`
  - `prior_mode`, `new_mode`, `queued_lines`, `oldest_queued_age_ms`,
    `entered_catch_up`

これらのイベントは、キューの圧力、選択されたドレイン動作、および時間経過に伴うモード遷移を示すことで、
表示遅延を説明することを意図している。
