# TUI の代替スクリーンとターミナルマルチプレクサ

## 概要

本ドキュメントでは、特に Zellij のようなターミナルマルチプレクサにおける Codex の代替スクリーンの扱いに関する設計判断について説明します。これは、フルスクリーン TUI の挙動とターミナルのスクロールバック履歴の保持との間にある根本的な衝突に対処するものです。

## 問題

### フルスクリーン TUI の利点

Codex の TUI はターミナルの **代替スクリーンバッファ** を使用して、クリーンなフルスクリーン体験を提供します。このアプローチにより、次のことが可能になります。

- ターミナルのスクロールバック履歴を汚さずにビューポート全体を使用できる
- チャットインターフェイス専用の環境を提供できる
- 他のターミナルアプリケーション（vim、tmux など）と同様の挙動になる

### Zellij における衝突

**Zellij** のようなターミナルマルチプレクサは xterm の仕様に厳密に従っており、その仕様では代替スクリーンバッファにはスクロールバックを持たせては**ならない**と定義されています。これはバグではなく、意図された設計です。

- **Zellij PR:** <https://github.com/zellij-org/zellij/pull/1032>
- **根拠:** xterm の仕様には、代替スクリーンモードではスクロールバックが許可されないことが明示されています
- **設定可否:** これは Zellij では設定変更できません。代替スクリーンモードでスクロールバックを有効にするオプションは存在しません

Zellij 上で Codex の TUI を使用する場合、ユーザーは会話履歴をスクロールバックできません。その理由は次のとおりです。

1. TUI が代替スクリーンモード（フルスクリーン）で動作している
2. Zellij は代替スクリーンバッファでスクロールバックを無効化している（xterm の仕様に従っているため）
3. その結果、通常のターミナルのスクロール操作では会話全体にアクセスできなくなる

## 解決策

Codex では、`config.toml` の `tui.alternate_screen` で制御される 3 つのモードによる**実用的な回避策**を実装しています。

### 1. `auto`（デフォルト）

- **動作:** ターミナルマルチプレクサを自動検出する
- **Zellij の場合:** 代替スクリーンモードを無効にする（インラインモードになり、スクロールバックが保持される）
- **それ以外の場合:** 代替スクリーンモードを有効にする（フルスクリーン体験）
- **理由:** 環境ごとに最適な UX を提供するため

### 2. `always`

- **動作:** 常に代替スクリーンモードを使用する（元の挙動）
- **ユースケース:** フルスクリーンを好み Zellij を使用していないユーザー、または何らかの回避策を見つけたユーザー

### 3. `never`

- **動作:** 代替スクリーンモードを一切使用しない（インラインモード）
- **ユースケース:** 常にスクロールバック履歴を保持したいユーザー
- **トレードオフ:** TUI の出力によってターミナルのスクロールバックが埋め尽くされる

## 実行時のオーバーライド

`--no-alt-screen` CLI フラグは、実行時に設定をオーバーライドできます。

```bash
codex --no-alt-screen
```

これにより、設定に関係なく TUI をインラインモードで実行できます。次のような用途に便利です。

- スクロールバックが重要な単発のセッション
- ターミナル関連の問題のデバッグ
- 代替スクリーンの挙動のテスト

## 実装の詳細

### 自動検出

`auto` モードは、`ZELLIJ` 環境変数を確認することで Zellij を検出します。

```rust
let terminal_info = codex_core::terminal::terminal_info();
!matches!(terminal_info.multiplexer, Some(Multiplexer::Zellij { .. }))
```

この検出は、`codex-rs/tui/src/lib.rs` 内のヘルパー関数 `determine_alt_screen_mode()` で行われます。

### 設定スキーマ

`AltScreenMode` enum は `codex-rs/protocol/src/config_types.rs` で定義されており、小文字の TOML にシリアライズされます。

```toml
[tui]
# オプション: auto, always, never
alternate_screen = "auto"
```

### なぜ Zellij では代替スクリーンを恒久的に無効化しないのか？

Zellij で常に無効化するのではなく `auto` 検出を使用している理由は次のとおりです。

1. 多くの Zellij ユーザーはスクロールバックを気にせず、フルスクリーンの体験を好む
2. 一部のユーザーは Zellij の内部で tmux を使用しており、マルチプレクサの多段構成になっている場合がある
3. 手動の設定変更を必要とせずにユーザーに選択肢を提供できる

## 今後の検討事項

### 検討した代替アプローチ

1. **TUI 内に独自のスクロールバックを実装する:** すべての履歴出力をバッファリングしてレンダリングする必要があり、大規模なアーキテクチャ変更が必要になる
2. **Zellij に設定オプションの追加を依頼する:** 現実的ではない — Zellij のメンテナは仕様に従うためにこの挙動を明示的に選択している
3. **無条件に代替スクリーンを無効化する:** Zellij 以外のユーザーの UX が低下する

### トランスクリプトページャ

Codex のトランスクリプトページャ（Ctrl+T で開きます）は、フルスクリーンモードでも会話履歴を確認するための代替手段を提供します。ただし、自然なスクロールバックほどシームレスではありません。

## 開発者向け

TUI のコードを変更する際は、次の点に注意してください。

- `determine_alt_screen_mode()` 関数がすべてのロジックをカプセル化している
- 設定は `config.tui_alternate_screen` にある
- CLI フラグは `cli.no_alt_screen` にある
- 挙動は `tui.set_alt_screen_enabled()` を通じて適用される

Codex 実行後にターミナルの状態に問題が発生した場合は、次のコマンドでターミナルを復元できます。

```bash
reset
```
