# TUI ストリームチャンク処理の検証プロセス

このドキュメントは、アダプティブなストリームチャンク処理と
アンチフラップの振る舞いを検証するために用いたプロセスを記録する。

## 範囲

目的は、ランタイムトレースから次の2つの性質を検証することにある。

- キュー負荷が高まったときに表示遅延が低減されていること
- モード遷移が急速なフラップではなく安定していること

## トレース対象

チャンク処理の観測用トレースは次から出力される:

- `codex_tui::streaming::commit_tick`

次の2種類のトレースメッセージを使用する:

- `stream chunking commit tick`
- `stream chunking mode transition`

## 実行時コマンド

chunking トレースを有効にして Codex を実行する:

```bash
RUST_LOG='codex_tui::streaming::commit_tick=trace,codex_tui=info,codex_core=info,codex_rmcp_client=info' \
  just codex --enable=responses_websockets
```

## ログ取得プロセス

ヒント: 単発の計測を行う場合は、`-c log_dir=...` を指定してログを新しいディレクトリに出力し、
セッションが混ざらないようにする。

1. `~/.codex/log/codex-tui.log` の現在のサイズを開始オフセットとして記録する。
2. 継続的にストリーム出力を生成するインタラクティブプロンプトを実行する。
3. 実行を停止する。
4. 記録したオフセット以降に書き込まれたログバイトのみを解析する。

これにより、以前のセッションと現在の計測ウィンドウが混ざるのを防げる。

## 評価するメトリクス

各計測ウィンドウについて:

- `commit_ticks`
- `mode_transitions`
- `smooth_ticks`
- `catchup_ticks`
- ドレインプランの分布 (`Single`, `Batch(n)`)
- キュー深度 (`max`, `p95`, `p99`)
- 最古のキューエントリの経過時間 (`max`, `p95`, `p99`)
- 急速な再エントリ回数:
  - `CatchUp -> Smooth` 遷移から1秒以内に発生した
    `Smooth -> CatchUp` 遷移の数

## 解釈

- 健全な振る舞い:
  - バックログをドレインしている間もキューの滞留時間が有界のままである
  - 総 tick 数に対して遷移回数が少ない
  - 急速な再エントリイベントはまれで、バーストの境界付近に局在している
- 退行した振る舞い:
  - 長い時間窓にわたって短い間隔でモードが繰り返しトグルする
  - smooth モード中にキュー滞留時間が持続的に増加する
  - バックログが減らないまま catch-up が長時間続く

## 実験履歴

このセクションでは、これまでに試した主要なチューニングパスを記録し、
今後の作業がそれらの結果の上に構築できるようにする。

- ベースライン
  - 50ms の commit tick で 1 行ずつ smooth にドレインしていた。
  - これにより従来と同様のペースは維持できたが、バックログが継続する状況では
    もたつきを感じることがあった。
- Pass 1: 即時 catch-up、ベースライン tick は不変
  - smooth モードのセマンティクスは維持しつつ、catch-up tick ごとに
    キュー済みバックログ全体をドレインするようにした。
  - 結果: キュー遅延はより速く減少したが、smooth モードのケイデンスが粗いままのため、
    体感される動きは依然として段階的に感じられることがあった。
- Pass 2: より高速なベースライン tick (25ms)
  - smooth モードのケイデンスが改善され、目に見えるステップ感が減少した。
  - 結果: 改善はしたが、描画のケイデンスとはまだ整合していなかった。
- Pass 3: フレームに整合したベースライン tick (~16.7ms)
  - ベースラインの commit ケイデンスをおよそ 60fps に設定した。
  - 結果: ヒステリシスと高速なバックログ収束を維持しつつ、
    体感される進行がより滑らかになった。
- Pass 4: より高いフレーム整合ベースライン tick (~8.3ms)
  - ベースラインの commit ケイデンスをおよそ 120fps に設定した。
  - 結果: 同じアダプティブ catch-up ポリシーの形状を保ったまま、
    smooth モードのステップ感をさらに低減した。

現在の状態は次を組み合わせたものになっている:

- `CatchUp` における即時の catch-up ドレイン
- モードの入出に対する安定性を確保するためのヒステリシス
- フレームに整合した smooth モードの commit ケイデンス (~8.3ms)

## 備考

- 検証はソース非依存であり、特定の上流プロバイダを名指しすることには依存していない。
- このプロセスは、既存のベースラインの smooth な振る舞いを意図的に維持しつつ、
  バースト/バックログ処理の振る舞いに焦点を当てている。
