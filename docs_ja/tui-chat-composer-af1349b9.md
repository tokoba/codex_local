# Chat Composer ステートマシン (TUI)

このメモは、`ChatComposer` の入力ステートマシンと、Windows ターミナル向けに追加されたペースト関連の挙動を文書化したものです。

主な実装:

- `codex-rs/tui/src/bottom_pane/chat_composer.rs`

ペーストバースト検出器:

- `codex-rs/tui/src/bottom_pane/paste_burst.rs`

## どの問題を解決しているか？

一部のターミナル（特に `crossterm` 経由の Windows）では、_ブラケット付きペースト_ が 1 回のペーストイベントとして安定して表面化しません。代わりに、複数行の内容をペーストすると、以下のようなキーイベントの高速な連なりとして現れることがあります:

- テキストに対しては `KeyCode::Char(..)`
- 改行に対しては `KeyCode::Enter`

コンポーザーがこれらのイベントを「通常のタイピング」として扱うと、次のようなことが起こり得ます:

- ペーストがまだストリーミング中に UI トグル（例: `?`）が誤ってトリガーされる
- `Enter` が届いたタイミングで、ペーストの途中でメッセージが送信されてしまう
- 入力済みプレフィックスをいったん描画してから、十分な文字数が溜まった時点でそれをペーストとして「再分類」し、フリッカーを起こす

解決策は、ペーストらしい _バースト_ を検出し、それらをバッファリングして 1 回の明示的な `handle_paste(String)` 呼び出しにまとめることです。

## 高レベルなステートマシン

`ChatComposer` は実質的に 2 つの小さなステートマシンを組み合わせています:

1. **UI モード**: どのポップアップ（もしあれば）がアクティブか。
   - `ActivePopup::None | Command | File | Skill`
2. **ペーストバースト**: 非ブラケット付きペーストの一時的な検出状態。
   - `PasteBurst` によって実装されています

### キーイベントのルーティング

`ChatComposer::handle_key_event` は `active_popup` に基づいてディスパッチします:

- ポップアップが表示されている場合、そのポップアップ専用のハンドラがキーを最初に処理します（ナビゲーション、選択、補完）。
- それ以外の場合、`handle_key_event_without_popup` が、送信（Enter）や履歴ナビゲーションなどのより高レベルなセマンティクスを処理します。
- キー処理の後に `sync_popups()` が実行され、ポップアップの表示状態やフィルターが最新のテキストとカーソル位置と整合するようにします。
- スラッシュコマンド名が補完され、その後ユーザーがスペースを入力すると、`/command` トークンはテキスト要素に昇格し、見た目が区別され、かつ要素全体が一括して編集されるようになります。

### 履歴ナビゲーション (↑/↓)

↑/↓ による呼び出しは `ChatComposerHistory` が処理し、2 つのソースを統合します:

- **永続履歴**（セッションをまたいで `~/.codex/history.jsonl` から取得される）: テキストのみ。テキスト要素の範囲や画像の添付は保持**しない**ため、これらのエントリを呼び出してもテキストだけが復元されます。
- **ローカル履歴**（現在のセッション）: テキスト要素、ローカル画像パス、リモート画像 URL を含む送信ペイロード全体を保存します。ローカルエントリを呼び出すと、プレースホルダや添付が再構成されます。

この区別により、ディスク上の履歴の後方互換性を保ちつつ、添付を永続化せずに、セッション内の編集に対してはよりリッチな履歴呼び出し体験を提供します。

## 再利用のための設定ゲーティング

`ChatComposer` は `ChatComposerConfig`
（`codex-rs/tui/src/bottom_pane/chat_composer.rs`）を通じて機能ゲーティングをサポートするようになりました。デフォルトコンフィグは現行のチャット動作を維持します。

フラグ:

- `popups_enabled`
- `slash_commands_enabled`
- `image_paste_enabled`

無効化された場合の主な影響:

- `popups_enabled` が `false` のとき、`sync_popups()` は `ActivePopup::None` を強制します。
- `slash_commands_enabled` が `false` のとき、コンポーザーは `/...` の入力をコマンドとして扱いません。
- `slash_commands_enabled` が `false` のとき、スラッシュコンテキストにおけるペーストバーストの例外処理が無効になります。
- `image_paste_enabled` が `false` のとき、ファイルパスペーストによる画像添付はスキップされます。
- `ChatWidget` は、選択されたモデルの `input_modalities` に基づいて、実行時に `image_paste_enabled` をトグルすることがあります。添付および送信のパスでもサポートを再チェックし、ドラフトを破棄するのではなく警告を出します。

組み込みスラッシュコマンドの利用可否は
`codex-rs/tui/src/bottom_pane/slash_commands.rs` に集約されており、コンポーザーとコマンドポップアップの両方で再利用されるため、ゲーティングの状態が同期したまま保たれます。

## 送信フロー (Enter/Tab)

送信パスはいくつかありますが、同じコアルールを共有しています。

steer モードが有効なときは、すでにタスクが実行中であれば `Tab` はキューイングを要求し、そうでなければ即座に送信します。このモードでは `Enter` は常に即座に送信します。入力が `!`（シェルコマンド）で始まる場合、`Tab` は送信しません。

### 通常の送信/キューイングパス

`handle_submission` は、送信とキューイングの両方で `prepare_submission_text` を呼び出します。このメソッドは次のことを行います:

1. 保留中のペーストプレースホルダを展開し、要素の範囲が最終的なテキストと整合するようにします。
2. 空白をトリムし、要素の範囲をトリム後のバッファに合わせて再ベースします。
3. トリミング後も残っているプレースホルダだけが送信されるよう、添付を間引きます。
4. 成功した場合は保留中のペーストをクリアし、最終テキストが空で添付もない場合は送信を抑止します。

同じ準備処理は、引数付きのスラッシュコマンド（たとえば `/plan` や `/review`）でも再利用されるため、引数を抽出するときにペーストされた内容やテキスト要素が保持されます。

コンポーザーはまた、テキストエリアの kill バッファを、表示されているドラフトとは別の編集状態として扱います。送信またはスラッシュコマンドのディスパッチによってテキストエリアがクリアされた後でも、直近の `Ctrl+K` のペイロードは `Ctrl+Y` で利用可能なままです。これにより、ユーザーがドラフトの一部を kill し、推論レベルの変更などのコンポーザーアクションを実行し、その後そのテキストをクリアされたドラフトに yank し直すといったフローをサポートします。

## リモート画像行（選択/削除フロー）

リモート画像の URL は、同じコンポーザーボックス内でテキストエリアの上に `[Image #N]` 行として表示されます。これらは添付行であり、編集可能なテキストエリアの内容ではありません。

- TUI はこれらの行を削除できますが、その前や間に文字を入力することはできません。
- テキストエリアのカーソル位置が `0` のときに `Up` を押すと、最後のリモート画像行が選択されます。
- 選択中は、`Up`/`Down` でリモート画像行間の選択を移動します。
- 最後の行で `Down` を押すと、リモート行の選択を抜けてテキストエリアの編集に戻ります。
- `Delete` または `Backspace` で選択中のリモート画像行を削除します。

画像の番号付けは統一されています:

- リモート画像行は常に `[Image #1]..[Image #M]` を占有します。
- ローカル添付画像プレースホルダは、そのオフセットの後から始まり（`[Image #M+1]..`）、続きます。
- リモート行を削除すると、ローカルプレースホルダに付け直しが行われ、番号付けが途切れず連続した状態に保たれます。

## 履歴ナビゲーション (Up/Down) とバックトラックのプレフィル

`ChatComposerHistory` は 2 種類の履歴を統合します:

- **永続履歴**（セッションをまたいで core からオンデマンド取得される）: テキストのみ。
- **ローカル履歴**（この UI セッション）: ドラフト状態全体。

ローカル履歴のエントリには次のものが含まれます:

- 生テキスト（プレースホルダを含む）
- プレースホルダ用の `TextElement` の範囲
- ローカル画像パス
- リモート画像 URL
- ドラフト用の保留中の大きなペーストペイロード

永続履歴のエントリはテキストだけを復元します。添付や保留中のペーストペイロードを再構成しないよう、意図的にそうなっています。

ドラフトが空でない場合、現在のテキストが最後に呼び出した履歴エントリと一致し、かつカーソルが境界（行頭または行末）にあるときにのみ、Up/Down のナビゲーションを履歴呼び出しとして扱います。これにより、シェルのような履歴トラバーサルを維持しつつ、複数行でのカーソル移動をそのまま保てます。

### ドラフトの復元 (Ctrl+C)

Ctrl+C はコンポーザーをクリアしますが、テキスト要素、ローカル画像パス、リモート画像 URL、保留中のペーストペイロードを含むドラフトの完全な状態をローカル履歴に退避します。直後に Up を押すと、そのドラフトが、画像プレースホルダや大きなペーストプレースホルダとそのペイロードも含めて復元されます。

### 送信済みメッセージの再呼び出し

送信が成功すると、ローカル履歴エントリには送信されたテキスト、要素範囲、ローカル画像パス、およびリモート画像URLが保存されます。送信中に保留中のペーストペイロードはクリアされるため、大きなペーストのプレースホルダーは記録される前に完全なテキストへ展開されます。つまり、次のようになります。

- 送信済みメッセージを Up/Down で再呼び出すと、リモート画像行に加えてローカル画像プレースホルダーが復元されます。
- 再呼び出されたエントリでは、一般的なシェル履歴の編集と同様に、カーソルは行末に配置されます。
- 再呼び出された送信履歴には大きなペーストのプレースホルダーは現れず、テキストは展開済みのペースト内容になります。

### Backtrack プリフィル

Backtrack の選択は、トランスクリプトから `UserHistoryCell` データを読み取ります。コンポーザーのプリフィルは、選択されたメッセージのテキスト要素、ローカル画像パス、リモート画像URLを再利用するようになったため、以前のユーザーメッセージにロールバックした際に、画像プレースホルダーと添付ファイルが再構成されます。

### 外部エディターでの編集

コンポーザーの内容が外部エディターから置き換えられた場合、コンポーザーはテキスト要素を再構築し、新しいテキスト内にプレースホルダーが残っている添付ファイルだけを保持します。その後、画像プレースホルダーは `[Image #M]..[Image #N]` に正規化されます。ここで `M` はリモート画像行の数の直後から開始し、編集後も添付ファイルの対応関係が一貫するようにします。

## ペーストバースト: コンセプトと前提

バースト検出器は意図的に保守的になっています。処理対象は「プレーン」な文字入力（Ctrl/Alt 修飾なし）のみです。それ以外の入力は、ショートカットが本来の意味を保つように、バーストウィンドウをフラッシュおよび／またはクリアします。

### 概念的な `PasteBurst` の状態

- **Idle**: バッファなし、保留中の文字なし。
- **Pending first char**（ASCII のみ）: ストリームがペーストであると判定された場合に文字を描画してすぐ削除することにならないよう、高速に入力された1文字をごく短時間だけ保持します。
- **Active buffer**: バーストがペースト的であると分類されたら、内容を `String` バッファに蓄積します。
- **Enter suppression window**: バーストの発生後しばらくの間は `Enter` を「改行」として扱い、小さな間隔があっても複数行ペーストがひとまとまりとして扱われるようにします。

### ASCII と非 ASCII（IME）入力

非 ASCII 文字はしばしば IME から入力され、正当な高速バーストとして到着することがあります。その場合に先頭の文字を保持してしまうと、入力が欠落したように感じられます。

そのため、コンポーザーは次のように区別します。

- **ASCII path**: 最初の高速な文字を保持することを許可します（`PasteBurst::on_plain_char`）。
- **non-ASCII path**: 先頭の文字を保持することは決して行いません（`PasteBurst::on_plain_char_no_hold`）。ただしバースト検出自体は行います。このパスでバーストが検出された場合、すでに挿入済みのプレフィックスをテキストエリアから遡って削除し、ペーストバッファに移動することがあります。

IME からのバーストをペーストと誤分類しないために、非 ASCII の遡及取得パスでは追加のヒューリスティック（`PasteBurst::decide_begin_buffer`）を実行し、遡って取得したプレフィックスが「ペーストっぽく」見えるかどうか（例: 空白を含む、十分な長さがある）を判定します。

### バースト検出の無効化

`ChatComposer` はエスケープハッチとして `disable_paste_burst` をサポートします。

有効化すると次のようになります。

- 新しい入力に対してはバースト検出器がバイパスされます（ちらつき抑制のための保持も、入力文字に対するバーストバッファリングの判定も行いません）。
- キーストリームは通常のタイプ入力として扱われます（スラッシュコマンドの通常の動作を含みます）。
- このフラグを有効にすると、保持／バッファリングされているバーストテキストは通常のペーストパス（`ChatComposer::handle_paste`）を通じてフラッシュされ、その後バーストタイミングおよび Enter 抑制ウィンドウがクリアされます。これにより、一時的なバースト状態が後続の入力に漏れないようにします。

### Enter の取り扱い

ペーストバーストのバッファリングがアクティブな場合、Enter は「メッセージを送信する」のではなく「バーストに `\n` を追加する」として扱われます。これにより、`Enter` キーイベントとして送出される複数行ペーストがペースト途中で送信されてしまうことを防ぎます。

またコンポーザーは、スラッシュコマンドコンテキスト（ポップアップが開いている、または最初の行が `/` で始まる場合）の内部では、バーストに基づく Enter 抑制を無効にし、コマンドのディスパッチが予測可能になるようにします。

## PasteBurst: イベントレベルの挙動（チートシート）

このセクションでは、`ChatComposer` が `PasteBurst` の判定をどのように解釈するかを詳述します。コードを「頭の中で実行」しなくても状態遷移を確認できるようにすることが目的です。

### プレーン ASCII `KeyCode::Char(c)`（Ctrl/Alt 修飾なし）

`ChatComposer::handle_input_basic` は `PasteBurst::on_plain_char(c, now)` を呼び出し、返された `CharDecision` に基づいて分岐します。

- `RetainFirstChar`: まだ `c` をテキストエリアに挿入し **ません**。UI の1ティック後に、`PasteBurst::flush_if_due` を介して通常のタイプ文字としてフラッシュされる場合があります。
- `BeginBufferFromPending`: 先頭の ASCII 文字はすでに保持／バッファリングされています。`PasteBurst::append_char_to_buffer` を通じて `c` をバッファに追加します。
- `BeginBuffer { retro_chars }`: すでに挿入されたプレフィックスを遡及的に取得しようとします。
  - `PasteBurst::decide_begin_buffer(now, before_cursor, retro_chars)` を呼び出します。
  - `Some(grab)` を返した場合は、テキストエリアから `grab.start_byte..cursor` を削除し、その後 `c` をバッファに追加します。
  - `None` を返した場合は、通常の挿入にフォールバックします。
- `BufferAppend`: アクティブなバッファに `c` を追加します。

### プレーン非 ASCII `KeyCode::Char(c)`（Ctrl/Alt 修飾なし）

`ChatComposer::handle_non_ascii_char` は少し異なるフローを使用します。

- まず `PasteBurst::flush_before_modified_input` によって、保留中の一時的な ASCII 状態（保持されている1文字の ASCII を含む）をフラッシュします。
- すでにバーストがアクティブな場合、`PasteBurst::try_append_char_if_active(c, now)` が `c` を直接追加します。
- それ以外の場合は `PasteBurst::on_plain_char_no_hold(now)` を呼び出します。
  - `BufferAppend`: アクティブなバッファに `c` を追加します。
  - `BeginBuffer { retro_chars }`: `decide_begin_buffer(..)` を実行し、バッファリングを開始する場合は、テキストエリアから遡って取得したプレフィックスを削除してから `c` を追加します。
  - `None`: `c` を通常どおりテキストエリアに挿入します。

このパスにおける追加の `decide_begin_buffer` ヒューリスティックは意図的なものです。IME からの入力は高速バーストとして到着することがあるため、IME の変換中入力をペーストと誤分類しないよう、プレフィックスが「ペーストっぽく」見える場合（空白を含む、ある程度以上の長さがあるなど）にのみ遡及取得を行います。

### `KeyCode::Enter`: 改行か送信か

「Enter が改行になる」メカニズムは2種類あります。

- **バーストコンテキスト内**（`paste_burst.is_active()`）: `append_newline_if_active(now)` は `\n` をバーストバッファに追加し、複数行ペーストが1つの明示的なペーストとしてバッファリングされ続けるようにします。
- **バーストの直後**（Enter 抑制ウィンドウ）: `newline_should_insert_instead_of_submit(now)` はテキストエリアに `\n` を挿入し、`extend_window(now)` を呼び出します。これにより、わずかに遅れた Enter でも「送信」ではなく「改行」として振る舞い続けます。

どちらのメカニズムも、スラッシュコマンドコンテキスト（コマンドポップアップがアクティブな場合、または最初の行が `/` で始まる場合）の内部では無効化され、コマンド編集中は Enter が通常どおり「送信／実行」の意味を持ち続けます。

### 非文字キー / Ctrl 付き修飾入力

非文字入力では、無関係なアクション間でバースト状態が漏れてはなりません。

- バッファリングされたバーストテキストが存在する場合、呼び出し側は `clear_window_after_non_char` を呼び出す前に（「Pitfalls worth calling out」を参照）、通常は `PasteBurst::flush_before_modified_input` を通じてそれをフラッシュする必要があります。
- `PasteBurst::clear_window_after_non_char` は「最近のバースト」ウィンドウをクリアするため、次のキー入力が誤って前のペーストにグルーピングされないようにします。

### 注意すべき落とし穴

- `PasteBurst::clear_window_after_non_char` は `last_plain_char_time` をクリアします。`buffer` が空でなく、かつ _まだフラッシュしていない_ 状態でこれを呼び出すと、`flush_if_due()` はタイムアウトを判定するためのタイムスタンプを持たなくなるため、バッファされたテキストがフラッシュされないままになる可能性があります。`clear_window_after_non_char` は「フラッシュ後に分類コンテキストを破棄する」ものとして扱い、「フラッシュ」として扱わないでください。
- `PasteBurst::flush_if_due` は厳密な `>` 比較を使用するため、テストや UI ティックは閾値を少なくとも 1ms 超える必要があります（`PasteBurst::recommended_flush_delay` を参照）。

## 注目すべき相互作用 / 不変条件

- composer はカーソル位置を使って `textarea.text()` を頻繁にスライスします。スライスを行うすべてのコードは、まずカーソルを UTF-8 の文字境界にクランプしなければなりません。
- ポップアップの表示やフィルタリングに影響を与えうるあらゆる変更（挿入、削除、バーストのフラッシュ、ペーストプレースホルダーの適用など）の後には、必ず `sync_popups()` を実行する必要があります。
- `?` によるショートカットオーバーレイのトグルは `!is_in_paste_burst()` によって制御されているため、ストリーミング中のペーストが UI モードを切り替えてしまうことはありません。
- メンションポップアップの選択には 2 種類のペイロードがあります。表示される `$name` テキストと、非表示の `mention_paths[name] -> canonical target` のリンクです。汎用の `set_text_content` パスは、新しいドラフト用にリンクを意図的にクリアします。一方、ブロック/中断された送信を再構成する復元パスでは、再試行時に元々選択されていたターゲットを保持できるよう、メンションを保持するセッターを使用する必要があります。

## 挙動を固定するテスト

`PasteBurst` のロジックは現在、`ChatComposer` の統合テストによって検証されています。

- `codex-rs/tui/src/bottom_pane/chat_composer.rs`
  - `non_ascii_burst_handles_newline`
  - `ascii_burst_treats_enter_as_newline`
  - `question_mark_does_not_toggle_during_paste_burst`
  - `burst_paste_fast_small_buffers_and_flushes_on_stop`
  - `burst_paste_fast_large_inserts_placeholder_on_flush`

このドキュメントでは、「クリアする前にフラッシュする」といった、専用の `PasteBurst` ユニットテストではまだ十分に固定されていない追加の契約事項についても言及しています。
