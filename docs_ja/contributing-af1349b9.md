## コントリビューション

**外部からのコントリビューションは招待制のみです**

現時点では、Codex チームは依頼していないコードのコントリビューションを受け付けていません。

新機能の提案や挙動の変更を提案したい場合は、提案内容を説明する issue を作成するか、既存の機能拡張リクエストに賛成票（upvote）を投じてください。新機能の優先度は、コミュニティからのフィードバック、当社のロードマップとの整合性、そして Codex のすべてのインターフェース（CLI、IDE 拡張、Web など）間での一貫性に基づいて決定されます。

バグに遭遇した場合は、バグレポートを作成するか、既存のレポートがすでにその問題をカバーしていないかを確認してください。もし協力したい場合は、issue スレッドで直接、解析結果、再現手順の詳細、根本原因の仮説、あるいは潜在的な修正のハイレベルな概要を共有するかたちでコントリビュートしていただくことを推奨します。

Codex チームは、次のような場合に外部コントリビューターにプルリクエスト（PR）の送信を依頼することがあります。

- 問題が十分に理解されている
- 提案されたアプローチがチームの意図する解決策と整合している
- その issue がインパクトが大きく、優先度も高いと判断されている

Codex チームのメンバーから明示的に招待されていないプルリクエストは、レビューされることなくクローズされます。

**外部からのコードコントリビューションを一般には受け付けていない理由**

過去には、Codex チームはバグ修正のための外部からのプルリクエストを受け付けていました。そのようなコミュニティからの労力と関与には感謝していましたが、このモデルはうまくスケールしませんでした。

多くのコントリビューションは、Codex の開発を導いているアーキテクチャ上のコンテキスト、システムレベルの制約、あるいは短期的なロードマップ上の考慮事項への十分な可視性がないまま行われていました。また、優先度が低い、あるいはごく一部のユーザーにしか影響しない問題にフォーカスしているものもありました。これらの PR をレビューし、修正を重ねることは、しばしば修正を直接実装するよりも多くの時間を要し、より優先度の高い作業から注意をそらすことになりました。

最も価値の高いコントリビューションは一貫して、問題領域について深い理解を示してくださるコミュニティメンバーからもたらされてきました。そのような専門知識は、詳細なバグレポート、解析、issue 内での設計ディスカッションといったかたちで、早い段階で共有されるときに最も役に立ちます。適切な解決策を見極めることこそが通常もっとも難しい部分であり、Codex 自身の助けを借りれば、その実装は比較的容易です。

これらの理由から、外部からのコントリビューションはディスカッション、解析、フィードバックにフォーカスし、コード変更は、明確に狙いを定めた招待が妥当な場合に限ることにしています。

### 開発ワークフロー

Codex チームメンバーから PR へのコントリビュートを招待された場合、推奨される開発ワークフローは次のとおりです。

- `main` から _トピックブランチ_ を作成します（例: `feat/interactive-prompt`）。
- 変更内容にはフォーカスを持たせてください。関連しない複数の修正は、別々の PR として作成してください。
- 変更によって lint の警告やテストの失敗が発生していないことを確認してください。

### 招待されたコードコントリビューションのためのガイドライン

1. **Issue から始めてください。** 新しい issue を作成するか、既存のディスカッションにコメントし、コードを書く前に解決策について合意できるようにします。
2. **テストを追加または更新してください。** バグ修正には、変更前には失敗し、変更後には成功するテストカバレッジを基本的に伴わせてください。100% のカバレッジは必須ではありませんが、有意なアサーションを目指してください。
3. **挙動をドキュメント化してください。** 変更がユーザー向けの挙動に影響する場合は、README、インラインヘルプ（`codex --help`）、または関連するサンプルプロジェクトを更新してください。
4. **コミットはアトミックに保ってください。** それぞれのコミットはコンパイルでき、テストが通る状態であるべきです。これにより、レビューやロールバックが容易になります。

### モデルメタデータの更新

変更によってモデルカタログやモデルメタデータ（`/models` のペイロード、プリセット、フィクスチャ）が更新される場合は、次の点に留意してください。

- 画像をサポートしないモデルには、`input_modalities` を明示的に設定してください。
- 互換性維持のためのデフォルトを念頭に置いてください。現在、`input_modalities` を省略すると、テキスト + 画像サポートを意味します。
- 画像を受け付けるクライアント側のインターフェース（例: TUI の貼り付け/添付）が、同じケイパビリティシグナルを利用していることを確認してください。
- サポートされていない画像に対する挙動と警告経路をカバーするテストを追加・更新してください。

### プルリクエストの作成（招待された場合のみ）

- PR テンプレート（または同等の情報）を埋めてください — **What? Why? How?** を記載します。
- Issue トラッカー上のバグ報告または機能拡張リクエストへのリンクを含めてください。
- **すべての**チェックをローカルで実行してください。ワークスペース全体と一貫性を保つため、ルートの `just` ヘルパーを使用してください: 触れたクレートに対して `just fmt`、`just fix -p <crate>` を実行し、該当するテスト（例: `cargo test -p codex-tui`、あるいは全面的なテストが必要な場合は `just test`）を実行します。ローカルで検出できたはずの CI の失敗はプロセスを遅らせます。
- 自分のブランチが `main` と同期しており、マージコンフリクトを解消していることを確認してください。
- PR がマージ可能な状態になったと判断した時点でのみ、PR を **Ready for review** に設定してください。

### レビュープロセス

1. 1 人のメンテナーが主レビュー担当としてアサインされます。
2. 招待された PR であっても、事前に議論・承認されていないスコープや振る舞いを導入している場合、PR をクローズすることがあります。
3. 変更をお願いすることがあります。個人的に受け止めないでください。私たちは皆さんの作業を尊重していますが、同時に一貫性と長期的な保守性も重視しています。
4. PR が基準を満たしているという合意が得られたら、メンテナーが squash-and-merge を行います。

### コミュニティの価値観

- **親切かつインクルーシブであること。** お互いを敬意をもって扱ってください。私たちは [Contributor Covenant](https://www.contributor-covenant.org/) に従っています。
- **善意を前提にすること。** 文章でのコミュニケーションは難しいものです — 相手に寛大である側に倒してください。
- **教え、学ぶこと。** 何か分かりにくい点に気づいたら、提案や明確化を添えて issue やディスカッションを開いてください。

### ヘルプを得るには

プロジェクトのセットアップで問題に直面した場合、アイデアへのフィードバックが欲しい場合、あるいは単に _こんにちは_ と言いたい場合でも、Discussion トピックを立てるか関連する issue に参加してください。喜んでお手伝いします。

一緒に Codex CLI を素晴らしいツールにしていきましょう。**楽しいハッキングを！** :rocket:

### コントリビューターライセンス契約（CLA）

すべてのコントリビューターは CLA に同意する **必要があります**。手続きは軽量です。

1. プルリクエストを作成します。
2. 次のコメントを貼り付けてください（以前に署名している場合は `recheck` と返信してください）。

   ```text
   I have read the CLA Document and I hereby sign the CLA
   ```

3. CLA-Assistant ボットがリポジトリ内にあなたの署名を記録し、ステータスチェックをパスとしてマークします。

特別な Git コマンド、メール添付、コミットフッターなどは不要です。

### セキュリティと責任ある AI

脆弱性を発見した場合や、モデルの出力に懸念がある場合は、**<security@openai.com>** までメールでご連絡ください。迅速に対応いたします。
